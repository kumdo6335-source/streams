# 스트림스(Streams) 교실 수학게임 웹앱 PRD

## 1. 개요
- **목적**: 초등학교 수업 시간(수학) 중 짧게 진행할 수 있는 보드게임 '스트림스'를 웹앱으로 구현
- **컨셉**: 교실 수업용, 수학 게임 명목
- **참가 방식**: 방장(교사)이 방 생성 → 참가자(학생)가 코드 입력 후 닉네임 등록 → 방장이 숫자를 하나씩 뽑으면 각자 20칸 판에 배치

## 2. 대상 사용자
| 역할 | 기기 | 화면 |
|---|---|---|
| 방장 | 교사 PC/TV (가정) | 진행 컨트롤 + 전체 현황 |
| 참가자 | 태블릿(가로) | 자신의 판 + 점수표 |

## 3. 게임 규칙 (원본 규칙 그대로)

### 3.1 타일 구성 (40개)
- 1~10: 각 1개 (10개)
- 11~19: 각 2개 (18개)
- 20~30: 각 1개 (11개)
- 조커: 1개

### 3.2 진행
- 방장이 "뽑기" 버튼을 클릭할 때마다 덱에서 타일 1개를 무작위로 뽑아 공개 (수동 진행)
- 모든 참가자가 현재 숫자를 자신의 20칸 중 빈 칸 하나에 배치해야 다음 뽑기 가능 (배치 전까지 방장 대기)
- 한 번 배치한 숫자는 이동 불가
- 총 20회 타일을 뽑으면(모든 칸이 채워지면) 게임 종료

### 3.3 점수 계산
- 20칸을 경로 순서대로 확인하며, 다음 숫자가 이전 숫자보다 크면 같은 구간으로 이어짐 / 작거나 같으면 구간이 끊기고 새 구간 시작
- 조커는 앞뒤 상관없이 항상 이어짐 (단, 조커 양옆이 모두 있을 때 조커가 두 숫자를 동시에 이어주진 못함 — 조커 앞 또는 뒤 중 한쪽으로만 해석)
- 각 구간 길이만큼 아래 점수표에서 점수를 찾아 전부 합산

### 3.4 점수표
| 개수 | 점수 | 개수 | 점수 |
|---|---|---|---|
| 1 | 0 | 11 | +30 |
| 2 | +1 | 12 | +35 |
| 3 | +3 | 13 | +40 |
| 4 | +5 | 14 | +50 |
| 5 | +7 | 15 | +60 |
| 6 | +9 | 16 | +70 |
| 7 | +11 | 17 | +85 |
| 8 | +15 | 18 | +100 |
| 9 | +20 | 19 | +150 |
| 10 | +25 | 20 | +300 |

## 4. 판(보드) 레이아웃
- 원본 실물 보드처럼 **구불구불한 냇물(지그재그) 모양**의 20칸 경로
- 경로 순서 예시 (상단 좌→우 → 우측 세로 → 하단 우→좌, 총 20칸):
  - 1~8번 칸: 상단, 좌→우
  - 9~12번 칸: 우측, 위→아래
  - 13~20번 칸: 하단, 우→좌
  - *(가정: 정확한 칸 수 분배는 8-4-8. 시각적 비율에 따라 구현 중 조정 가능)*
- 빈 공간(경로 안쪽)에 점수표 배치
- 학생 화면에도 점수표 표시 (전략적 판단 지원)

## 5. 화면별 요구사항

### 5.1 방장(교사) 화면
- 방 생성 버튼 → 4자리 방 코드 자동 생성 및 큰 글씨로 표시 (학생들이 보고 입력하도록)
- 참가자 목록(닉네임) 실시간 표시
- "뽑기" 버튼: 클릭 시 덱에서 타일 1개 무작위 추첨 → 현재 숫자 큰 글씨로 표시
- 모든 참가자가 현재 숫자를 배치하기 전까지 "뽑기" 버튼 비활성화(또는 대기 안내), 배치 현황(몇 명 중 몇 명 완료) 표시
- "취소(되돌리기)" 버튼: 방금 뽑은 타일을 덱으로 되돌리고 직전 상태로 복구 (단, 이미 학생들이 배치를 완료해 다음 숫자로 넘어간 뒤에는 취소 불가 — 직전 1수만 취소 가능)
- 게임 종료 시 전체 순위표(닉네임+점수, 내림차순) 표시

### 5.2 참가자(학생) 화면
- 코드 입력 화면 → 닉네임 입력 화면 → 대기 화면(방장이 게임 시작할 때까지)
- 게임 중: 현재 뽑힌 숫자 표시 + 자신의 20칸 판(빈 칸 클릭하여 배치) + 점수표 상시 표시
- 배치 완료 후 다음 숫자가 나올 때까지 "배치 완료, 대기 중" 상태 표시
- 게임 진행 중에는 다른 학생의 판/점수 비공개 (자기 판만 보임)
- 게임 종료 시 전체 순위표 + 본인 최종 점수/구간 내역 표시

## 6. 데이터 모델 (Firestore)
```
rooms (collection)
  {roomCode} (document, 4자리)
    status: "waiting" | "playing" | "ended"
    deck: (number|"J")[]        // 셔플된 잔여 타일
    drawHistory: (number|"J")[] // 지금까지 뽑힌 순서
    currentTile: number | "J" | null
    createdAt: timestamp
    lastDrawAt: timestamp

    players (subcollection)
      {nickname 또는 playerId} (document)
        nickname: string
        board: (number|"J"|null)[20]  // 경로 순서대로 20칸
        placedCount: number
        joinedAt: timestamp
```
- 방장 화면: `rooms/{code}` 문서 + `players` 서브컬렉션 전체를 `onSnapshot`으로 구독
- 학생 화면: `rooms/{code}` 문서(현재 숫자 등 공통 정보) + 본인 `players/{nickname}` 문서만 구독
- "모든 참가자 배치 완료" 판정: 방장 화면에서 players 서브컬렉션을 집계하여 `placedCount === drawHistory.length` 인원 수 확인

## 7. 기술 구조
- **플랫폼**: 독립 웹앱 (Firebase Hosting 배포 전제), 순수 HTML/JS + Firebase SDK (v9 modular)
- **DB**: Firestore, `onSnapshot` 실시간 구독 (폴링 아님, 진짜 실시간)
- **인증**: 없음. 코드 소유 여부로만 접근 제한 (아래 8번 보안 항목 참고)
- **구현 환경**: Antigravity(별도 IDE/에이전트)에서 이 PRD를 기반으로 직접 구현 예정 — 본 문서는 Claude가 아닌 다른 도구에서의 구현을 위한 스펙 문서로 사용
- **배포**: 사용자가 본인 Firebase 프로젝트에 직접 배포 (Firebase 콘솔 설정, `firebaseConfig` 값 입력, `firebase deploy` 실행 포함)
- **필요 입력값**: 사용자의 기존 Firebase 프로젝트 `firebaseConfig` 객체 (apiKey, projectId 등) — 별도 설정 파일(`firebase-config.js`)에 입력하는 형태로 분리 권장

## 8. 보안 규칙 (Firestore Security Rules)
- 로그인이 없으므로 사용자 신원 검증 불가 → **코드(문서 ID)를 아는 사람만 접근 가능**하다는 전제로 규칙 설계 (가정)
- 예: `rooms/{code}` 문서는 누구나 읽기/쓰기 가능하지만, 목록 조회(list)는 금지 → 4자리 코드를 모르면 어떤 방이 존재하는지 알 수 없음
- 참고: 이 방식은 "완전한 보안"은 아니며 교실 내 단발성 게임 용도로 충분한 수준 (가정). 더 엄격한 보안이 필요하면 익명 인증 도입을 다시 고려해야 함

## 9. 확정된 결정 사항 요약
- 게임 규칙: 원본 그대로 (타일 40개, 조커 포함, 점수표 그대로)
- 판 형태: 냇물 모양 지그재그 20칸 + 빈 공간에 점수표
- 플랫폼: 독립 웹앱, Firebase(기존 프로젝트) + Firestore 실시간 구독
- 인증: 로그인 없이 보안 규칙(코드 비공개 + list 금지)으로만 제한
- 배포: 사용자가 Antigravity에서 구현 후 직접 Firebase 배포
- 방 코드: 4자리 자동 생성
- 타일 추첨: 방장 수동 클릭
- 점수 공개 시점: 게임 중엔 비공개, 종료 후 전체 공개
- 학생 화면: 점수표 항상 표시
- 취소(되돌리기) 기능: 필요 (직전 1수만)
- 재대결(같은 방에서 다시하기): 불필요 — 새 방 생성으로 대체

## 10. 가정 사항 (미확정 — 구현 중 기본값으로 처리, 필요시 수정 요청)
- 학생이 태블릿 새로고침/재접속 시 같은 닉네임으로 자동 복귀 (가정: 필요)
- 경로 칸 수 분배는 8-4-8 (상단-우측-하단) 비율로 시각 구현 (가정)
- 닉네임 중복 시 자동으로 번호 접미사 부여 (예: 민수 → 민수(2)) (가정)
- 참가 인원 수 제한 없음 (가정: 학급 인원 기준 최대 40명 내외로 충분)
- Firestore 보안 규칙은 인증 없이 코드 기반 접근 제한으로 처리 (가정, 8번 참고)

## 11. 범위 외 (Out of scope)
- 같은 방에서 재시작/재대결 기능
- 로그인/회원 관리
- 게임 기록 영구 저장(DB), 통계 대시보드
